# 2026-06-12 — Tailscale Serve + Control UI Fix

## Session: Patticus + Talos

### Problem
- Control UI was requiring auth token paste every browser open — clunky and frustrating
- `gateway.controlUi.allowInsecureAuth: true` with `dangerouslyDisableDeviceAuth: true` was a brittle workaround

### Solution: Tailscale Serve
**Config changes to `openclaw.json`:**
- `gateway.tailscale.mode: "off"` → `"serve"`
- `gateway.auth.allowTailscale: true` (enables Tailscale identity header auth)
- `gateway.controlUi.allowInsecureAuth: true` → `false`
- `gateway.controlUi.dangerouslyDisableDeviceAuth: true` → `false`
- Updated `allowedOrigins` to HTTPS Tailscale hostname: `https://serenity.tail4695cd.ts.net`

**Tailscale Serve setup (user ran in terminal):**
```bash
sudo tailscale set --operator=patrick
sudo tailscale serve / http://127.0.0.1:18789
```

Now `https://serenity.tail4695cd.ts.net/` proxies to the gateway on loopback. No token pasting — Tailscale identity headers handle auth automatically. HTTPS = WebCrypto works, no "insecure context" issues.

### Side Effect: Avatar Reset
- Switching from `http://192.168.1.155:18789/` to `https://serenity.tail4695cd.ts.net/` changed browser origin
- Browser-local avatar storage is per-origin, so the old upload vanished from the new URL
- Re-uploaded `avatars/talos-avatar.png` at the new origin — restored ✅

### Result
- Control UI: smooth, no token paste, HTTPS, working avatar
- Tailscale Serve: active for root path `/` → port 18789
- `/mission` route (port 18080) preserved

### Follow-ups
- Monitor for any other Control UI quirks with the new setup
- Consider setting `ui.assistant.avatar` in config if avatar resets become annoying across origin changes
